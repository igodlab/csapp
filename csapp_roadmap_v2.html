Why §3.9.3 was added
Bank conflicts in shared memory (PMPP 6.4, 15.6) arise when threads access addresses that map to the same bank. The address arithmetic depends on struct size and alignment — exactly what §3.9.3 teaches. Without it, you're memorizing bank conflict rules rather than deriving them.
PTX mirrors this mental model exactly — virtual registers, predicate registers instead of branches, explicit memory addressing. Needed to understand register pressure warnings in Nsight Compute and why register spilling hurts occupancy. §3.9.3 now also provides the struct layout reasoning required for shared memory optimization.

Why §4.5.8 was added
§4.5.5 covers hazards; §4.5.8 covers the hardware that resolves them. The GPU answer to pipeline stalls is warp switching — but "the GPU avoids stalls by switching warps" is a slogan until you understand what problem it's solving. §4.5.8 is four pages and provides that frame before PMPP 4.6.
GPU SMs are deeply pipelined. Latency hiding via warp switching is the GPU answer to pipeline hazards — you need the CPU pipeline model first to appreciate why the GPU solution works. §4.5.8 (new) provides the stall-resolution context that makes PMPP 4.6 mechanistically clear, not just descriptively true.

Why §7.12 was added
7.10–7.11 (included in v1) cover dynamic linking but leave PIC unexplained — which makes the runtime linking model incomplete. PIC is the mechanism that allows CUDA kernels to be loaded at arbitrary device addresses. Omitting it while including 7.10–7.11 was an inconsistency.
CUDA compilation produces device + host object files the linker stitches together. Symbol resolution explains non-obvious nvcc linking failures. PIC (new) completes the model: CUDA kernels are position-independent code loaded into the device's virtual address space at runtime, which is exactly the mechanism §7.12 describes.

Why this was downgraded from CRITICAL
The CUDA synchronization model (__syncthreads, __threadfence, warp-level primitives, C++ atomics with scope/ordering) has no POSIX analog. CSAPP Ch. 12 teaches pthread_create, mutex, condition variables — machinery that doesn't exist in CUDA. The concepts (race conditions, memory visibility, mutual exclusion) transfer; the API does not. The retained sections cover exactly the transferable concepts. PMPP Ch. 9–11 covers the GPU-native synchronization model far more directly.
Concepts transfer; API does not. You need to understand race conditions and memory visibility before reasoning about atomic operations in PMPP Ch. 9 (histogram) and barriers in PMPP Ch. 10–11 (reduction/scan). But the specific POSIX machinery is dead weight. Skim the conceptual sections; skip the pthread implementation sections entirely.

Roofline Model — Williams, Waterman & Patterson (2009)
Priority · read before PMPP Ch. 5
The primary analytical framework of PMPP Ch. 5–6 (arithmetic intensity, compute- vs memory-boundedness, speed-of-light analysis). CSAPP's CPE model is the CPU analog but doesn't cover roofline. This is the highest-priority gap.
Dao et al. — FlashAttention (2022)
Priority · read alongside PMPP §20.5
PMPP §20.5 covers flash attention pedagogically but summarizes rather than derives the online softmax correctness argument. The original paper is load-bearing for understanding why the tiled recomputation is correct — not just supplementary context.
NVIDIA CUDA Programming Guide — SIMT Architecture
Read alongside PMPP Ch. 4
The GPU SIMT execution model (warps, divergence, warp scheduling) has no CSAPP analog. The CUDA guide's SIMT sections provide hardware-accurate framing that PMPP Ch. 4 assumes but doesn't fully derive. Free, always current.
NVIDIA Hopper Architecture Whitepaper
Read before PMPP Ch. 15 + 20
Tensor cores are a fundamentally different compute unit from scalar FPUs. Neither CSAPP nor Micikevicius covers the hardware mechanism. The whitepaper explains why bf16 behaves differently than fp16 on Hopper vs Ampere — required for PMPP Ch. 15's tensor core and software pipelining coverage.
Micikevicius et al. — Mixed Precision Training (2018)
Read before PMPP Appendix A + Ch. 20
Already in your plan. Pairs with CSAPP 2.4 (IEEE 754, rounding) and PMPP Appendix A. Explains loss scaling, fp16 accumulation strategy, and why certain operations must stay in fp32. Problem 2.90 (new) builds the precision-loss intuition this paper relies on implicitly.
Patterson & Hennessy — I/O chapter (COD, ARM ed.)
Read before PMPP Ch. 23 if needed
PCIe and NVLink bandwidth arithmetic for multi-GPU reasoning. Only becomes critical if you engage seriously with PMPP Ch. 23 (multi-GPU programming). Lower priority than the items above.
